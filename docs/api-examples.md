# Calling open-runo from JavaScript, TypeScript, or any other language

`open-runo-router`/`open-runo-gateway` speak plain JSON over HTTP -- there
is no Rust-specific transport, IDL, or SDK requirement. Anything that can
do `fetch()` (or an HTTP client in any language) can call this API
directly. This page is for a caller building a separate frontend --
vanilla HTML/CSS/JS, TypeScript, a Bootstrap-based admin panel, or a
backend in another language entirely -- who wants the same
"no hand-maintained types, no drift" experience Rust callers get from
`open-runo-api-types` (see `CLAUDE.md` HANDOFF, 2026-07-11).

The project's own frontend (`apps/desktop-wasm`) is Rust-to-WebAssembly
by design (see `CLAUDE.md`'s "フロントエンド" section) -- this doc is not
about reviving a TypeScript/Bootstrap *product* frontend, it's about
making the REST API itself pleasant to call from **whatever stack you're
already building your own client in**.

---

## 1. No API key setup required

Every route except `/health`/`/healthz` requires an `X-Api-Key` header,
but you never have to generate, configure, or paste one in by hand:
`POST /api/keys/self-issue` hands back a working `developer`-role key
(24h TTL) with no auth of its own. Fetch one on page load, cache it, and
transparently re-issue on a `401` -- this is exactly what
`apps/desktop-wasm/src/api.rs` and `open-runo-cli` both do.

```js
const BASE_URL = "http://localhost:8080"; // same-origin in production
let apiKey = localStorage.getItem("open-runo-api-key");

async function ensureApiKey() {
  if (apiKey) return apiKey;
  const res = await fetch(`${BASE_URL}/api/keys/self-issue`, { method: "POST" });
  const body = await res.json();
  apiKey = body.api_key;
  localStorage.setItem("open-runo-api-key", apiKey);
  return apiKey;
}

async function callApi(path, options = {}) {
  const key = await ensureApiKey();
  let res = await fetch(`${BASE_URL}${path}`, {
    ...options,
    headers: { ...options.headers, "X-Api-Key": key, "Content-Type": "application/json" },
  });
  if (res.status === 401) {
    // Key expired/revoked server-side -- drop it and retry once.
    apiKey = null;
    localStorage.removeItem("open-runo-api-key");
    const fresh = await ensureApiKey();
    res = await fetch(`${BASE_URL}${path}`, {
      ...options,
      headers: { ...options.headers, "X-Api-Key": fresh, "Content-Type": "application/json" },
    });
  }
  return res;
}
```

## 2. Handling errors: request IDs and rate limits

Every response carries an `X-Request-Id` header (generated by the server,
or echoed back if you send your own) -- include it when you report a bug,
so whoever reads the server logs can find your exact request:

```js
const res = await callApi("/api/federation/status");
if (!res.ok) {
  const requestId = res.headers.get("x-request-id");
  throw new Error(`request failed (${res.status}), request-id: ${requestId}`);
}
```

`429 Too Many Requests` responses carry a standard `Retry-After` header
*and* a JSON body shaped like `open-runo-api-types::RateLimitedResponse`
(see the generated `RateLimitedResponse` schema in
`GET /api/openapi.json`'s `components.schemas`):

```json
{ "error": "rate limit exceeded, see retry_after_secs", "retry_after_secs": 42 }
```

```js
if (res.status === 429) {
  const { retry_after_secs } = await res.json();
  console.warn(`rate limited, retry in ${retry_after_secs}s`);
}
```

## 3. CORS

`with_cors` (see `crates/open-runo-router/src/middleware_hyper.rs`) sends
CORS headers on every response and answers `OPTIONS` preflights directly,
so a browser-based client on a *different* origin (e.g. a Bootstrap admin
panel served from its own dev server) can call this API without a
same-origin restriction. By default any origin is allowed; set
`OPEN_RUNO_CORS_ALLOWED_ORIGINS` (comma-separated) to restrict it in
production. `X-Api-Key` and `Content-Type` are both allowed request
headers.

## 4. Generating TypeScript types from the live spec

`GET /api/openapi.json` is a real OpenAPI 3.0 document, and (as of
2026-07-11) `components.schemas` for the schema-registry and federation
endpoints is *generated* directly from the server's own Rust structs via
`schemars` (see `crates/open-runo-router/src/openapi.rs`) -- not
hand-written, so it can't drift from what the server actually sends.
Point any OpenAPI-based codegen tool at it. For TypeScript, the
zero-install option is
[`openapi-typescript`](https://www.npmjs.com/package/openapi-typescript):

```bash
# with a router already running on :8080
npx openapi-typescript http://localhost:8080/api/openapi.json -o open-runo.d.ts
```

That produces real, field-level types for the covered endpoints, e.g.:

```ts
import type { components } from "./open-runo.d.ts";

type SchemaVersion = components["schemas"]["SchemaVersion"];
// { id: string; namespace: string; service_name: string; sdl: string;
//   stage: string; created_at: string }

async function registerSchema(
  body: components["schemas"]["RegisterSchemaRequest"]
): Promise<SchemaVersion> {
  const res = await callApi("/api/schemas", { method: "POST", body: JSON.stringify(body) });
  if (!res.ok) throw new Error(`register failed: ${res.status}`);
  return res.json();
}
```

**Coverage note**: as of this writing, `components.schemas` covers the
schema registry (`SchemaVersion`, `RegisterSchemaRequest`,
`SchemaHistoryResponse`) and federation (`FederationStatusResponse`,
`RateLimitedResponse`) endpoints -- the ones with a shared
`open-runo-api-types` Rust type behind them. The other ~25 REST endpoints
(DB CRUD, SCIM, persisted queries, cache/backup) still have
`description`-only responses in the spec; extending `open-runo-api-types`
to cover them (and regenerating their schemas the same way) is tracked as
a follow-up in `CLAUDE.md`'s HANDOFF.

## 5. A minimal HTML + Bootstrap example

No build step, no framework -- just `fetch()` against the API from a
plain page, styled with Bootstrap's CDN like any other static site:

```html
<!doctype html>
<html>
<head>
  <link href="https://cdn.jsdelivr.net/npm/bootstrap@5/dist/css/bootstrap.min.css" rel="stylesheet">
</head>
<body class="container py-4">
  <h1>open-runo schema search</h1>
  <input id="service" class="form-control mb-2" placeholder="service name">
  <button id="search" class="btn btn-primary mb-3">Search history</button>
  <pre id="result" class="bg-light p-3"></pre>

  <script>
    const BASE_URL = "http://localhost:8080";
    // ... ensureApiKey()/callApi() from section 1 ...

    document.getElementById("search").addEventListener("click", async () => {
      const service = document.getElementById("service").value;
      const res = await callApi(`/api/schemas/${encodeURIComponent(service)}/history`);
      document.getElementById("result").textContent = JSON.stringify(await res.json(), null, 2);
    });
  </script>
</body>
</html>
```

Because of CORS (section 3), this page works whether it's served by
`open-runo-router` itself, a separate static file server, or opened
directly from disk -- as long as `BASE_URL` points at a reachable
instance.

## 6. GraphQL

The GraphQL Federation endpoint (`open-runo-gateway`'s
`POST /graphql`, GraphiQL playground at `GET /graphql`) is
Apollo/Relay/urql-compatible standard GraphQL over HTTP -- any GraphQL
client library in any language works unmodified. See `docs/federation.md`
for the schema composition model.
